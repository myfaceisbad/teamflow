## 技術的負債・改善

| 項目 | 現状 | 改善方向 |
|------|------|---------|
| 状態管理 | API直接fetch | TanStack Query のキャッシュ活用を徹底 |
| エラーハンドリング | try/catch + toast | グローバルエラーバウンダリ + 構造化エラー |
| 型安全性 | 一部 `any` 使用 | Prisma 型の完全活用 + Zod バリデーション |
| API 設計 | 個別ルート | tRPC or REST 規約の統一 |
| CSS | Tailwind直書き | デザイントークンの体系化 |
| デプロイ | ローカルのみ | Vercel / Docker + CI/CD パイプライン |

## 成功指標（KPI）

| 指標 | 目標値 |
|------|--------|
| マーケットプレイス応募率 | タスクの 30%+ がメンバー自ら応募 |
| ワークロード可視化の利用率 | 週次で 80%+ のメンバーが体感負荷を入力 |
| タスク配分の偏り | チーム内の負荷標準偏差を 20% 削減 |
| メンバー満足度 | 「担当タスクへの納得感」スコア 4.0/5.0 以上 |
| バーンアウト検知 | 高負荷 3日連続を 90%+ の精度で検知 |

---

> **TeamFlow の核心**: 「マネージャーがアサインする」から「メンバーが選べる」へのパラダイムシフト。
> このコンセプトを軸に、段階的に機能を拡充していく。
